# Principle 0: Write for the reader

Documentation exists to help a specific reader understand or do something with confidence.

Start every documentation task by asking:

- Who is this for?
- What should they understand or be able to do after reading it?
- What is the simplest structure that helps them get there?

## Core Rules

- Optimize for reader understanding, not template compliance.
- Prefer clear structure over rigid structure.
- Use examples, tables, code blocks, trees, and admonitions when they genuinely improve comprehension.
- Keep pages lean, but not so compressed that they become cryptic.
- Remove repetition when two sections do the same job.

## Practical Standard

Good documentation should feel:

- accurate
- easy to scan
- easy to trust
- easy to act on
